README.md (v2 — Architecture Expanded) # Voice MCP OS — Distributed Voice Execution System  ## Overview  Voice MCP OS is a mobile-first distributed voice operating system that combines:  - real-time voice communication (calls / chat / voice messages) - AI execution layer (MCP tools + LLM reasoning) - system control layer (Android OS automation) - presence layer (friends / sessions / availability) - always-on audio pipeline (optional BLE-bound activation)  It is NOT a voice assistant.  It is a **voice-driven execution + communication OS layer**.  ---  ## Core Concept  > "Voice is the primary system bus for human-device interaction"  Everything in the system is a voice event: - commands - calls - messages - system actions - AI queries  ---  ## System Architecture  ### 1. Audio Core Layer - microphone capture (low-latency streaming) - VAD segmentation - optional RNNoise / WebRTC APM - audio session tagging (call / command / idle)  ---  ### 2. Communication Layer (CRITICAL LAYER)  This is a first-class subsystem.  Supports:  #### Real-time protocols: - WebRTC (primary) - QUIC (low-latency transport / fallback) - Mumble (self-hosted group voice) - TeamSpeak (legacy interoperability)  #### Features: - peer-to-peer voice calls - group channels - low-latency voice streams - fallback routing between protocols - adaptive codec selection  ---  ### 3. Presence & Social Layer  System includes built-in social graph:  #### Core entities: - user profile - friend list - online presence state - device state (mobile / desktop / wearable)  #### Actions: - add / remove friend - send message - read message - send voice message - initiate voice call - join voice session  Presence states: - ONLINE - BUSY (call / AI session) - IDLE - OFFLINE - BLE-LINKED ACTIVE MODE  ---  ### 4. AI Execution Layer (MCP Runtime)  LLM is NOT chat.  It is a **command interpreter + planner**.  #### MCP tools: - search - summarize - notes - reminders - calendar - messaging automation - system control - custom plugins  Output format: ```json {   "intent": "...",   "action": "...",   "params": {},   "confidence": 0.0-1.0 } 5. Android OS Control Layer  Direct device automation:  open/close apps notification control system settings media control accessibility-driven automation background services  This layer is MCP-driven.  6. BLE Activation Layer (Context Binding)  Critical system trigger layer:  BLE scan for known devices trusted device proximity = system activation disconnect = auto suspend voice OS used for: earbuds smartwatch car system wearable trigger  Behavior:  BLE CONNECT → activate voice OS runtime BLE DISCONNECT → shutdown AI session + audio pipeline 7. Session Model  System runs in concurrent sessions:  COMMUNICATION SESSION (calls / voice chat) AI EXECUTION SESSION (MCP tools) SYSTEM CONTROL SESSION (OS actions)  All sessions share audio input stream but differ in routing.  8. Response Layer  Outputs:  TTS (voice feedback) UI overlay (optional) chat log (secondary) call audio stream (primary) Execution Model  User voice input is routed:  Voice Input →  VAD → Session classifier → Communication OR AI OR System FSM → MCP execution OR voice stream → response output Non-Goals chatbot UI text-first interaction generic AI conversation system standalone LLM product Success Criteria seamless switching between call and AI execution <300ms routing decision latency (local prototype relaxed) stable voice session concurrency BLE-driven lifecycle control works reliably MCP tools fully externalizable Target Platforms Android (primary runtime) desktop (development reference) wearable (BLE trigger only) Philosophy  "The phone is not an app container anymore.
